# AutoGLM 测试用例编写规范

## 生成前硬性规则

生成用例前必须先阅读本节。本节优先级高于后文所有模板。任何违反本节的用例都视为不合格，必须重写。

### 禁止标题填充模板

严禁把“用例标题”或“功能树叶子名称”直接塞进固定模板中生成步骤。除非该标题本身就是 App 中真实可见的 UI 文案，否则步骤中禁止出现下列句式：

```text
打开“[完整用例标题]”功能
进入“[完整用例标题]”页面
进入“[完整用例标题]”面板
查找“[完整用例标题]”入口
找不到“[完整用例标题]”入口
执行“[完整用例标题]”核心交互
检查“[完整用例标题]”最终状态
页面停留在“[完整用例标题]”相关页面
```

不合格示例：

```text
打开“我的页账号信息加载”功能。
打开“视频点赞按钮状态验证”功能。
打开“首页目标手表卡片与同步状态”功能。
执行“首页下拉刷新与异常状态”核心交互。
```

合格写法必须把标题翻译成真实承载页面、真实控件和真实可观察状态：

```text
进入“我的”页，检查头像、昵称、账号状态、会员/登录状态、加载失败提示或空状态。
进入任一视频播放页，点击点赞按钮，检查点赞状态、数量变化、登录提示或失败原因。
停留在首页/智能设备首页，检查目标手表卡片中的手表名称、连接状态、电量、同步状态或添加设备入口。
在首页/智能设备首页执行一次下拉刷新，检查刷新完成、加载中、网络错误、空态或重试入口。
```

### 必须先做功能点语义分类

每个叶子功能在生成步骤前必须先归类，且步骤必须符合该类型：

| 类型 | 判断标准 | 步骤写法 |
| --- | --- | --- |
| 页面入口类 | App 中真实存在入口或页面，例如设置、关于、隐私政策、搜索 | 进入承载模块 -> 点击真实入口文案/图标 -> 检查目标页 |
| 页面状态类 | 标题包含状态、加载、展示、同步、异常、卡片、数据展示 | 进入承载页面 -> 检查状态/空态/失败提示，不找同名入口 |
| 页面内控件交互类 | 点赞、收藏、分享、评论、播放/暂停、进度、开关、二次确认 | 进入控件所在页面 -> 操作真实控件 -> 检查状态变化/弹窗/提示 |
| 设备能力/环境依赖类 | 血压、血糖、压力、体温、NFC、微信、翻译、眼镜媒体等依赖设备能力 | 进入设备功能页 -> 支持则验证；不支持/无入口/未连接/暂无数据则记录明确限制结果 |
| 敏感流程类 | 绑定手机号、修改密码、退出登录、注销、解绑、恢复出厂、删除、支付、发布、发送 | 只到入口、验证页、说明页或二次确认弹窗；不执行最终提交 |

### 强制自检失败即重写

生成完成后，必须逐条检查。如果任一用例命中以下规则，必须重写该用例，不允许保留：

- 标题包含 `状态`、`加载`、`展示`、`同步`、`异常`、`按钮状态验证`，但步骤中出现 `打开“完整标题”`、`进入“完整标题”页面`、`找不到“完整标题”入口`。
- 标题包含 `点赞`、`收藏`、`分享`、`评论`、`播放暂停`、`进度控制`，但步骤中没有先进入 `视频播放页`、`视频详情页` 或 `任一可播放视频`。
- 标题包含 `我的页账号信息加载`、`个人资料展示`，但步骤中没有进入 `我的` 页后检查头像、昵称、账号状态、登录状态、加载失败提示或空状态。
- 标题包含 `首页目标手表卡片`、`首页目标眼镜卡片`，但步骤中要求打开同名功能页，而不是在首页检查设备卡片。
- 标题包含 `首页下拉刷新`，但步骤中要求打开同名入口，而不是在首页执行下拉刷新。
- 标题包含血压、血糖、压力、体温、梅脱、NFC、微信、翻译、眼镜拍照、眼镜相册等设备能力，但没有写明“不支持/无入口/未连接/暂无数据/权限缺失”为明确限制结果。
- 标题包含退出登录、修改密码、绑定手机号、注销、解绑、恢复出厂、删除、支付、发布、发送，但步骤要求点击最终确认、提交或真实发送。

生成文件时可以保留完整覆盖数量，但不能为了套模板牺牲步骤真实性。

## 核心原则

1. 写目标，不只写点击步骤。
2. 写状态，不依赖坐标。
3. 写允许行为，也写禁止行为。
4. 写通过条件，也写失败条件。
5. 明确遇到弹窗、登录态、加载慢、异常提示时怎么处理。
6. 敏感数据使用占位符或环境变量，不在用例中写明文密码、验证码、身份证、支付信息。
7. 用例应能独立执行，不能依赖上一条用例留下的隐含状态。
8. 测试结果以最终可观察状态为准，而不是以模型自述为准。
9. 执行步骤必须按 1、2、3、4 编号，主流程顺序必须清楚。
10. 对于设备类，要针对不同设备生成对应的测试用例，并且写清楚前置条件：不同类型设备连接下如何切换设备、未连接设备时如何连接等；设备类用例应优先从首页设备卡片或首页设备入口开始操作，首页不可达时再进入设备页。
11. 对于设备类，如何切换或连接设备应该写在前置条件或通用提示中，不应作为单独测试用例；设备连接后允许的每一个功能都应单独生成测试用例，例如微信电话、微信回复、扫码、图片回复、一键翻译、Wearfit 翻译、面对面翻译、同声传译必须分别成例。每个功能用例必须包含真实可执行的操作步骤，验证功能页面、权限、数据加载、核心交互或明确的限制提示，不能只检查入口是否存在。测试数据可使用占位符或 mock，音频、视频、图片、文件等素材类数据可以只写占位符并说明后续提供。
12. 生成的测试用例文件，应该在文件开头总结所有测试用例，给出数目和标题等。
13. 单条测试用例的标题必须使用二级标题，格式为 `## [用例ID] 用例标题`。一级标题只用于整份文件标题，不用于单条用例。
14. 用例 ID 必须以 `TC-WEARFIT` 开头，并且最后一段必须标识规则类型。规则类型只能使用不含连字符的单词：普通用例使用 `normal`，需要电脑播放音频配合手机操作的用例使用 `audio`。用例 ID 中的模块段只能使用固定一级模块编码：`LOGIN`（登录）、`MINE`（我的）、`HOME`（首页）、`VIDEO`（视频）、`GLASSES`（眼镜）、`WATCH`（手表）。同一个模块编码必须始终对应同一个中文一级模块，不能把子功能名写进模块段。
15. 生成用例前必须明确本次覆盖范围，并告知用户将包含的设备类型和 App 功能模块（最好使用选择列表形式）。默认设备类型包括手表、眼镜；默认 App 功能模块包括首页、我的、视频；登录等必要链路可作为前置或支撑模块。这里的“模块”只表示一级覆盖范围，固定为登录、我的、首页、视频、眼镜、手表；测试用例文件中的“模块”列和“关联模块”字段也必须只写这 6 个一级模块之一。模块不表示每个模块只生成少量抽样用例；进入生成阶段后，必须继续拆解每个模块内的具体功能点，并为每一个可测试功能点分别生成独立用例。默认范围生成时，必须先按当前产品信息、代码入口、页面路由、资源文案、设备能力分组和用户已说明范围推导功能树，再按功能树叶子节点逐项生成，不得只按大类概括生成。设备模块必须按第 11 条继续拆到设备能力级别；视频、首页、我的等 App 模块同样要按入口、列表、详情、搜索、设置、账号安全等具体功能拆分。具体功能点写在用例标题、测试目标、步骤或功能描述中，不要写进“模块”列或“关联模块”字段。用例应尽量按真实操作顺序组织。生成前应列出拟生成的设备类型、功能模块、模块内功能点和顺序；如果用户已经明确要求按默认范围生成，可以直接生成，不必再次询问。
16. 每个执行步骤除了写“目标状态”，还必须单独增加一行“所在 Activity”。如果代码中能确认 Activity，应写完整类名；如果无法确认，应写“所在 Activity：待确认”。
17. 测试用例解析必须以二级标题作为唯一用例起始标记，格式为 `## TC-WEARFIT-[模块]-[序号]-[规则类型] 用例标题`。`---` 只允许作为视觉分隔线，不作为解析依据。一个完整用例块从该二级标题开始，到下一个符合格式的二级标题前结束；如果没有下一个用例标题，则一直到文件结尾。
18. 执行步骤必须能够被机器解析。每个步骤必须以行首编号 `1. `、`2. `、`3. ` 开始；一个步骤块从该编号行开始，到下一个编号步骤或下一个三级标题前结束。步骤内部字段必须独立成行，至少包含 `执行条件：`、`目标状态：`、`已满足处理：`、`失败处理：`、`所在 Activity：`。
19. 默认每条用例执行前，工程 runner 已经完成 App 重置并进入 Wearfit Pro 首页/智能设备首页。普通功能用例不得把“打开 App”“点击首页”“进入首页”“切换到首页”“返回首页”“返回初始页”“回到首页/智能设备首页”“关闭重开 App”写成状态恢复步骤。只有当用例本身验证首页导航、底部 Tab 切换、返回逻辑、页面恢复、App 启动、重启或冷启动时，才允许写这些动作，并且标题和测试目标必须明确说明验证的是该能力。普通功能用例结束时停留在目标功能完成后的合理页面即可，不强制返回首页。
20. 生成用例时必须先判断每个功能点的真实形态，再写步骤。功能点只能归入下列四类之一：页面入口、页面状态、页面内控件交互、设备能力/环境依赖能力。不同类型的步骤写法不同，不能把所有功能点都写成“打开 xxx 功能”。
21. 如果功能点是“状态验证”“数据加载”“卡片展示”“同步状态”“异常状态”“按钮状态验证”，它通常不是独立入口。步骤必须写成“进入承载页面后检查状态”，不得写成“打开 xxx状态验证入口”或“点击 xxx数据加载功能”。
22. 如果功能点是“点赞、收藏、分享、评论、播放/暂停、进度控制、开关切换、清理缓存、二次确认”等控件交互，步骤必须先进入真实承载页，再操作真实控件。不得假设存在同名页面或同名入口。
23. 如果功能点依赖具体设备能力，例如血压、血糖、压力、体温、梅脱/有效活动量、NFC、微信回复、翻译、眼镜相册、眼镜拍照等，前置条件和步骤必须写明“入口可见且设备支持时验证；入口不可见、设备不支持、未连接或暂无数据时记录明确限制提示并停止，不判为 App 失败”。除非用例标题明确是“设备不支持时应失败”，否则不要让模型无限查找不存在的入口。
24. 敏感流程必须停在安全边界。绑定手机号、修改密码、退出登录、注销账号、解绑设备、恢复出厂设置、删除数据、支付、发布、举报、真实发送消息等功能可以生成用例，但步骤只能验证入口、说明页、确认弹窗、验证码/密码验证页或明确限制提示；不得要求模型真正提交、退出、注销、解绑、删除、支付或发送。

推荐解析正则：

```regex
^##\s+(TC-WEARFIT-[A-Za-z0-9]+-\d{3}-(?:normal|audio))\s+(.+)$
```

推荐步骤解析规则：

```regex
^(\d+)\.\s+(.+)$
```

解析说明：

- 只在 `### 执行步骤` 到下一个 `###` 标题之间解析步骤。
- 每个步骤从行首编号开始，到下一个编号步骤或下一个三级标题前结束。
- 步骤编号必须连续递增。
- 每个步骤块必须能解析出 `执行条件`、`目标状态`、`已满足处理`、`失败处理`、`所在 Activity` 五个字段。
- 字段值可以是自然语言，但字段名必须固定，不能写成同义词，例如不要把 `已满足处理` 写成 `如果已满足`。

## 默认范围覆盖充分性要求

当用户要求“默认范围”“默认测试用例”或未额外指定范围时，必须按功能树而不是按模块大类生成。规范不得固定写死具体业务功能清单；生成者应根据当前可用的产品信息、代码入口、页面路由、资源文案、设备能力分组和用户已说明的范围，先整理“待覆盖功能树”，再为功能树的每个叶子功能生成独立用例。

默认范围的覆盖边界如下：

- 一级模块固定为登录、首页、我的、视频、眼镜、手表。
- 默认设备类型为手表、眼镜。
- 登录只覆盖进入 App 所需主链路和登录态恢复，不扩展到注册、找回密码、注销等高风险或非默认链路，除非用户明确要求。
- 首页、我的、视频必须覆盖可见一级入口、列表/信息流、详情/二级页、搜索、设置、编辑、提交/保存、权限/空态/异常提示等不同类型能力，不能只生成“页面加载”类用例。
- 我的模块必须把账号信息、设置页、账号与安全、绑定手机号、修改密码、退出登录二次确认、帮助与反馈、关于页、隐私政策、单位设置等不同页面或独立入口拆开生成；不能用“账号安全页面查看”或“个人资料展示”这类大用例合并多个入口。
- 我的模块不能只覆盖“打开页面”这一层；对于同一页面中存在可独立验证的状态查看、资料修改、设置切换、权限查看、反馈提交、版本/协议查看等核心交互时，也必须拆成独立用例。页面入口用例验证页面可达，核心交互用例验证页面内的具体业务动作或状态结果。
- 我的模块中账号信息加载、个人资料展示等属于状态类，应写成“进入我的页后检查头像、昵称、账号、会员/登录状态、加载失败提示或空状态”，不得写成同名入口。绑定手机号、修改密码、退出登录、注销账号等属于敏感流程，只验证入口、验证页或二次确认弹窗，不执行最终提交。
- 首页模块中首页主框架、设备卡片、健康数据卡片、Feed 卡片、下拉刷新、异常状态等大多属于状态类或页面内交互类，应在首页/智能设备首页直接检查或操作，不得写成“打开首页下拉刷新与异常状态入口”。
- 视频模块中点赞、收藏、分享、评论、播放/暂停、进度控制等属于视频播放页或视频详情页内控件交互，必须先进入任一可播放视频，再操作目标控件，不得生成同名功能页或同名入口。
- 手表、眼镜必须按设备功能菜单或设备能力分组继续拆分到叶子功能；健康、表盘、通知、提醒、二维码/NFC、媒体、微信、翻译、查找、导航、AI、设置、固件、电量、设备信息、未连接限制等能力不能被合并成大类用例。
- 如果一个大类下面存在多个独立入口、独立页面、独立开关、独立数据类型或独立交互，例如不同健康指标、不同提醒类型、不同翻译模式、不同二维码类型、不同媒体类型、不同眼镜设置页，必须分别生成用例。
- 对设备功能，连接、切换和未连接处理应写入前置条件或功能步骤中，不应代替功能本身；但未连接下受限功能的明确提示可以作为独立异常用例。
- 对设备能力不确定的功能，不允许因为“可能不支持”而删除用例。应保留该用例，并在前置条件、执行条件、目标状态、预期结果中写明：支持时验证目标页面或核心交互；不支持、无入口、未连接、暂无数据或权限缺失时记录明确限制结果并停止。
- 对音频、视频、图片、文件等素材类功能，测试数据可使用 mock 占位符；用例仍必须执行真实功能流程，不得只检查入口存在。
- 眼镜相关默认优先级为 P2；手表核心连接和同步为 P0，其余默认 P1；登录主链路为 P0；首页主框架和设备入口为 P0；其他 App 功能按影响范围使用 P1 或 P2。

生成默认范围用例前，必须在文件开头列出本次推导出的功能树和用例数量。功能树应体现“模块 -> 功能分组 -> 叶子功能”的拆分思路。生成后应自检：

- 是否存在把多个叶子功能合成一条用例的情况。
- 是否存在只验证入口、不验证页面状态或核心交互的情况。
- 是否每个设备功能都包含目标设备已连接、多设备切换、未连接连接方式或明确限制提示。
- 是否把状态类功能写成了承载页检查，而不是同名入口。
- 是否把控件交互类功能写成了“进入承载页 -> 操作控件 -> 检查结果”，而不是同名页面。
- 是否把设备能力不确定功能保留下来，并定义了“不支持/无入口/暂无数据/未连接/权限缺失”的可接受结果。
- 是否把敏感流程停在入口、验证页或二次确认弹窗，没有要求执行最终提交。
- 是否所有音频类功能都使用 `audio` 规则类型，并包含语音输入和结果检查。
- 是否所有步骤都包含 `执行条件`、`目标状态`、`已满足处理`、`失败处理`、`所在 Activity`。

## Runner 初始状态约束

当前工程 runner 会在每个用例执行前重置 App，并默认进入 Wearfit Pro 首页/智能设备首页。因此生成测试用例时必须遵守：

- 每条用例的前置条件必须写明：“工程 runner 会在每个用例执行前重置 App，并默认进入 Wearfit Pro 首页/智能设备首页。”
- 普通功能用例的第 1 步应直接从首页/智能设备首页进入目标模块、目标设备功能页或目标功能。例如：`从首页/智能设备首页进入 ${WATCH_A_NAME} 手表功能页。`
- 普通功能用例不要写“打开 Wearfit Pro”“关闭重开 App”“点击首页”“进入首页”“切换到首页”“返回首页”“回到初始页”“返回手表功能页”等状态恢复步骤。
- 普通功能用例如果需要进入手表或眼镜详情页，必须从首页/智能设备首页进入目标设备功能页，不能假设 runner 已经进入设备详情页。
- 普通功能用例结束时停留在目标功能页面、结果页、设置页、弹窗页或明确失败/限制提示页即可，不要为了恢复状态而返回首页。
- 如果用例验证的是首页导航、底部 Tab 切换、返回逻辑、页面恢复能力、App 启动、重启或冷启动，可以包含进入首页、返回首页、打开 App 或重启 App 等步骤；但用例标题、测试目标和步骤目标状态必须明确说明正在验证这些能力。
- 删除冗余首页/恢复步骤后，步骤编号必须重新从 1 连续递增。

推荐普通功能用例起始步骤：

```text
1. 从首页/智能设备首页进入 ${WATCH_A_NAME} 手表功能页。
   执行条件：工程 runner 已重置 App，当前停留在 Wearfit Pro 首页/智能设备首页，且首页设备卡片或设备入口可见时执行。
   目标状态：进入 ${WATCH_A_NAME} 手表功能页，页面显示目标手表名称、连接状态或功能列表。
   已满足处理：如果当前已经在 ${WATCH_A_NAME} 手表功能页，本步骤视为通过。
   失败处理：如果无法从首页/智能设备首页进入目标手表功能页，停止并记录“目标手表功能页不可达”。
   所在 Activity：待确认
```

不推荐普通功能用例写法：

```text
1. 打开 Wearfit Pro。
2. 点击底部导航中的“首页”。
3. 进入目标功能。
4. 返回首页。
```

上述步骤把 runner 已经完成的状态恢复写进了普通功能用例，会增加执行噪声并降低用例独立性。

## 功能点语义与步骤生成规则

生成用例时，不能只把功能标题机械套成“打开 xxx 功能”。必须先判断功能点的真实形态，再选择步骤模板。此规则用于保证不丢功能点，同时提升 9B 模型执行成功率。

### 页面入口类

页面入口类指 App 中真实存在的可点击入口、菜单项、Tab、列表项或设置项，例如“设置页打开”“关于页打开”“隐私政策页打开”“视频搜索入口打开”“表盘市场列表”。

推荐步骤结构：

```text
1. 从首页/智能设备首页进入目标一级模块。
   执行条件：当前停留在 Wearfit Pro 首页/智能设备首页，且底部导航或模块入口可见时执行。
   目标状态：进入目标一级模块页面，页面出现该模块标题、列表、设置入口或核心内容。
   已满足处理：如果当前已经在目标一级模块页面，本步骤视为通过。
   失败处理：如果目标一级模块入口不可见，停止并记录“目标模块入口不可见”。
   所在 Activity：待确认

2. 点击真实入口文案或真实控件进入目标页面。
   执行条件：当前在承载目标入口的页面，且目标入口可见时执行。
   目标状态：进入目标页面，页面出现标题、说明、列表、空状态、权限提示或加载失败原因。
   已满足处理：如果当前已经在目标页面，本步骤视为通过。
   失败处理：如果目标入口不可见，可以在当前页面合理滚动查找一次；仍不可见则停止并记录“目标入口不可见”。
   所在 Activity：待确认
```

禁止写法：

```text
2. 打开“设置页打开”功能。
2. 点击“账号信息加载”入口。
```

标题中带“打开”的用例，也必须写真实入口，例如“我的页右上角设置按钮”“视频页搜索入口”“手表功能页中的表盘市场”，不要把完整标题当作 UI 文案。

### 页面状态类

页面状态类指检查页面已经展示的状态，而不是进入一个新功能，例如“首页目标手表卡片与同步状态”“我的页账号信息加载”“首页下拉刷新与异常状态”“视频播放异常与重试入口”。

推荐步骤结构：

```text
1. 进入状态所在的承载页面。
   执行条件：当前停留在 Wearfit Pro 首页/智能设备首页，且承载页面入口可见时执行。
   目标状态：进入或停留在承载页面，页面出现目标卡片、列表、信息流或账号区域。
   已满足处理：如果当前已经在承载页面，本步骤视为通过。
   失败处理：如果承载页面不可达，停止并记录“承载页面不可达”。
   所在 Activity：待确认

2. 检查目标状态。
   执行条件：当前在承载页面时执行。
   目标状态：页面展示目标状态、正常数据、空状态、加载中、失败提示、重试入口或权限提示中的任一种明确结果。
   已满足处理：如果已经看到上述任一明确结果，本步骤通过并结束当前用例。
   失败处理：如果页面白屏、黑屏、崩溃、无任何明确内容或长时间无响应，停止并记录具体异常。
   所在 Activity：待确认
```

禁止写法：

```text
1. 从首页打开“首页目标手表卡片与同步状态”。
2. 打开“我的页账号信息加载”功能。
3. 查找“首页下拉刷新与异常状态”入口。
```

这些标题描述的是要观察的状态，不是页面入口。生成时应把它们落到真实承载页面上。

### 页面内控件交互类

页面内控件交互类指必须先进入某个页面，再操作页面上的按钮、图标、开关或输入框，例如“视频点赞按钮状态验证”“视频收藏按钮状态验证”“打开视频分享面板”“视频播放暂停与进度控制”“退出登录二次确认”“缓存清理提示查看”。

推荐步骤结构：

```text
1. 进入控件所在的承载页面。
   执行条件：当前停留在 Wearfit Pro 首页/智能设备首页，且模块入口可见时执行。
   目标状态：进入包含目标控件的页面，例如视频播放页、我的设置页、账号安全页或手表功能页。
   已满足处理：如果当前已经在包含目标控件的页面，本步骤视为通过。
   失败处理：如果承载页面不可达，停止并记录“目标控件承载页面不可达”。
   所在 Activity：待确认

2. 操作目标控件。
   执行条件：当前页面存在目标按钮、图标、开关、输入框或二次确认入口时执行。
   目标状态：控件状态发生可观察变化，或出现面板、弹窗、提示、错误原因、权限提示或重试入口。
   已满足处理：如果目标面板、弹窗或状态变化已经出现，本步骤视为通过。
   失败处理：如果目标控件不可见，可以在当前承载页合理查找一次；仍不可见则停止并记录“目标控件不可见”。
   所在 Activity：待确认

3. 检查操作结果。
   执行条件：目标控件已操作或已经显示目标状态时执行。
   目标状态：页面没有崩溃、白屏、黑屏或长时间无响应，且展示控件状态、面板、提示或明确失败原因。
   已满足处理：如果已经满足目标状态，本步骤通过并结束当前用例。
   失败处理：如果没有任何反馈且重试一次后仍无响应，停止并记录“目标控件无响应”。
   所在 Activity：待确认
```

禁止写法：

```text
2. 打开“视频点赞按钮状态验证”功能。
2. 进入“视频收藏按钮状态验证”页面。
2. 查找“视频播放暂停与进度控制”入口。
```

点赞、收藏、分享、播放/暂停不是同名页面；它们是视频播放页或详情页里的控件。

### 设备能力与环境依赖类

设备能力类指功能是否存在依赖具体手表/眼镜型号、固件版本、连接状态、权限或真实硬件能力，例如血压、血糖、压力、体温、梅脱/有效活动量、NFC、微信回复、翻译、眼镜相册、眼镜拍照、眼镜直播等。

前置条件必须写清楚：

```text
- 设备状态：目标设备已连接或可连接；如果当前设备不支持本功能，页面出现“暂不支持”、入口不可见、暂无数据、未连接提示或权限提示时，记录为可接受限制结果并停止，不判为 App 失败。
```

推荐步骤结构：

```text
1. 从首页/智能设备首页进入目标设备功能页。
   执行条件：当前停留在 Wearfit Pro 首页/智能设备首页，且目标设备卡片或设备入口可见时执行。
   目标状态：进入目标设备功能页，页面显示目标设备名称、连接状态、功能列表或未连接提示。
   已满足处理：如果当前已经在目标设备功能页，本步骤视为通过。
   失败处理：如果目标设备卡片不可见或设备功能页不可达，停止并记录“目标设备功能页不可达”。
   所在 Activity：待确认

2. 查找并打开目标设备能力入口。
   执行条件：当前在目标设备功能页，且设备已连接或页面允许查看功能列表时执行。
   目标状态：进入目标能力页面，或看到设备不支持、暂无数据、未连接、权限缺失、入口不可见等明确限制结果。
   已满足处理：如果已经在目标能力页面，或已经看到明确限制结果，本步骤视为通过并进入结果检查。
   失败处理：如果合理滚动查找一次后仍无入口，也没有任何明确提示，停止并记录“目标能力入口不可见，疑似设备不支持或入口命名不一致”。
   所在 Activity：待确认

3. 检查目标能力页面或限制结果。
   执行条件：当前已经进入目标能力页面，或已经看到明确限制结果时执行。
   目标状态：页面展示目标数据、空状态、权限提示、未连接提示、不支持提示、暂无数据、失败原因或重试入口。
   已满足处理：如果看到上述任一明确结果，本步骤通过并结束当前用例。
   失败处理：如果页面白屏、黑屏、崩溃、卡死或无任何明确内容，停止并记录具体异常。
   所在 Activity：待确认
```

禁止写法：

```text
3. 一直滚动直到找到血糖健康数据。
3. 如果没有血压入口则继续查找其他页面。
3. 找不到压力数据就判定失败。
```

设备能力不存在或当前型号不支持时，不应把 App 判为失败。失败只针对崩溃、白屏、无响应、入口点击无反馈且没有明确限制提示等 App 行为异常。

### 敏感流程与高风险动作类

敏感流程包括绑定手机号、修改密码、退出登录、注销账号、解绑设备、恢复出厂设置、删除数据、支付、发布内容、举报、真实发送消息、上传隐私文件等。

这类用例允许生成，但必须停在安全边界：

```text
目标状态：进入目标入口、说明页、验证码页、密码验证页、二次确认弹窗、权限提示或明确限制提示；不执行最终提交。
禁止行为：不要点击最终确认、提交、退出、注销、解绑、删除、支付、发送或发布按钮。
```

推荐写法：

```text
3. 打开退出登录二次确认弹窗。
   执行条件：当前在账号与安全页或设置页，且“退出登录”入口可见时执行。
   目标状态：页面出现退出登录二次确认弹窗，包含“取消”和“确认/退出”等按钮。
   已满足处理：如果确认弹窗已经出现，本步骤视为通过。
   失败处理：如果入口不可见，停止并记录“退出登录入口不可见”。
   所在 Activity：待确认

4. 检查并关闭二次确认弹窗。
   执行条件：当前显示退出登录二次确认弹窗时执行。
   目标状态：确认弹窗内容可读，并且点击“取消”后回到原页面。
   已满足处理：如果弹窗已展示且可读，本步骤通过；如需关闭，只能点击“取消”。
   失败处理：如果误触最终确认导致退出登录，记录为触发禁止行为。
   所在 Activity：待确认
```

禁止写法：

```text
4. 点击确认退出登录。
4. 完成修改密码。
4. 提交绑定手机号。
4. 确认恢复出厂设置。
```

## 基于代码生成用例时的要求

如果用例由人或模型阅读代码后生成，应优先从代码中提取以下信息，再写成自然语言步骤：

- 应用包名、目标 activity、页面类名或路由名。
- 页面入口，例如底部 tab、菜单项、按钮文案。
- 关键控件文案，例如“登录”、“添加设备”、“保存”。
- 关键输入框提示，例如“手机号”、“请输入密码”。
- 可用于判断结果的页面文案、resource-id、content-desc。
- 业务前置条件，例如需要登录、需要蓝牙、需要定位权限、需要真实设备。
- 高风险动作，例如支付、删除、解绑、注销、修改资料。

代码信息只能作为增强提示，不应把用例写成脆弱的坐标脚本。

推荐：

```text
1. 点击底部导航中的“设备”。
   执行条件：当前在 Wearfit Pro 首页或可见底部导航时执行。
   目标状态：进入设备页，页面中出现“添加设备”或设备列表。
   已满足处理：如果当前已经在设备页，本步骤视为通过。
   失败处理：如果底部导航不可见，停止并记录“设备入口不可见”。
   所在 Activity：待确认
```

不推荐：

```text
1. 点击坐标 520,1810。
```

如果代码中能确认 activity 或 resource-id，可以写进预期结果的可观察状态中：

```text
可观察状态：

- 当前 activity 应为设备页相关 activity。
- 页面中应出现 resource-id 为 `com.example:id/add_device` 的入口，或出现文案“添加设备”。
```

## 音频相关步骤写法

涉及语音、录音、翻译、同声传译等能力时，测试步骤仍然要按用户可理解的功能流程来写，不要把规则机制写进用例。用例只描述“要完成一次语音输入/翻译/识别流程”，音频播放、长按、录音等环境配合由测试环境处理。

音频相关步骤要避免以下说法：

```text
等待自动规则执行“播放测试音频并长按识别键”。
音频素材无需提供，后续由测试环境处理。
不点击开始录音或翻译。
```

这些说法会让模型不知道当前到底要不要执行语音流程，也容易把规则当成测试终点。

推荐写法：

```text
3. 在一键翻译页面执行一次语音输入。
   执行条件：当前已经进入一键翻译页面，并且页面存在录音、按住说话或语音输入入口时执行。
   目标状态：页面进入录音、识别或翻译处理中状态；测试环境会提供标准音频输入。
   已满足处理：如果页面已经处于录音、识别或处理中状态，本步骤视为通过。
   失败处理：如果找不到语音输入入口，停止并记录“一键翻译语音入口不可见”。
   所在 Activity：待确认

4. 等待翻译结果返回。
   执行条件：当前页面已经完成一次语音输入，且处于识别、翻译或处理中状态时执行。
   目标状态：页面展示翻译文本、识别文本、失败提示或明确的重试入口。
   已满足处理：如果已经看到翻译文本、识别文本、失败提示或重试入口，本步骤视为通过。
   失败处理：如果等待超时仍无结果或无失败提示，停止并记录“翻译结果未返回”。
   所在 Activity：待确认

5. 检查翻译结果页面。
   执行条件：始终执行。
   目标状态：页面没有崩溃、白屏或卡死，且能看到翻译结果或明确失败原因。
   已满足处理：如果已经满足目标状态，本步骤通过并结束当前用例。
```

## 字段说明

### 用例ID

用例 ID 必须稳定，并且必须和二级标题写在同一行。固定格式：

```text
TC-WEARFIT-[模块]-[序号]-[规则类型]
```

规则类型说明：

- `normal`：普通用例，不需要特殊外部动作。
- `audio`：需要电脑播放音频，并配合手机长按、录音或语音识别的用例。

示例：

```text

TC-WEARFIT-WATCH-002-normal
TC-WEARFIT-WATCH-053-audio
```

单条用例标题格式必须是：

```markdown
## TC-WEARFIT-LOGIN-001-normal 手机号密码登录后进入主页
```

不要频繁修改 ID。用例管理和缺陷关联会依赖它。

### 优先级

- P0：核心链路，失败会阻断主流程，例如启动、登录、进入首页、核心设备连接。
- P1：重要功能，失败影响主要体验，但不一定阻断所有流程。
- P2：边缘场景、兼容性、文案、低频入口。

### 测试目标

测试目标回答“为什么执行这个用例”。不要只写“测试登录功能”，要写清楚验证点。

推荐：

```text

```

不推荐：

```text

```

- 是否需要蓝牙、定位、通知权限。
- 是否需要外接设备或后端环境。

对于登录态，用例必须明确：

```text
如果已经登录并进入主页，可以直接检查主页状态并判定通过，不需要退出登录后重新登录。
```

或者：


### 执行步骤

执行步骤是直接喂给 agent 的核心内容。必须按 1、2、3、4 编号，表达清楚主流程顺序。

每个步骤必须使用固定字段。字段名必须独立成行，不能合并成一段话：

- 动作：agent 需要做什么。
- 执行条件：什么状态下执行本步骤；默认写“始终执行”。
- 目标状态：做完后应该看到什么。
- 已满足处理：如果当前已经满足目标状态，应如何处理。
- 失败处理：如果无法完成，应记录什么原因，是否允许重试。
- 所在 Activity：执行该步骤时预期或实际所在的 Activity。

```text

   所在 Activity：`com.wakeup.howear.view.MainActivity`
```

不要写跨步骤跳转：

```text

```

```text

已满足处理：如果当前已经在主页，本步骤跳过，不要退出登录。
```

```text
   失败处理：如果无法判断登录状态，停止并记录“无法判断登录状态”。

   失败处理：如果未进入主页且没有明确错误提示，停止并记录“未进入主页”。
   所在 Activity：`com.wakeup.howear.view.MainActivity`
   停留在主页，不要继续进入无关功能。
```

不推荐：


原因是所有动作挤在一段里，agent 容易漏步骤，也不利于执行过程按步骤回放和排查。

每一步都要避免依赖坐标或相对位置。

不推荐：

```text

2. 点击第二个输入框。
3. 点击蓝色按钮。
```

登录、启动、重启、冷启动类用例推荐：

```text

2. 输入测试手机号。
3. 点击文本为“密码”或占位提示为“请输入密码”的输入框。

5. 点击文本为“登录”的按钮。
```

原因是 UI 位置和样式容易变化，9B 模型也可能对“第二个”理解不稳定。

### 预期结果

预期结果描述最终应观察到的状态。尽量写成可检查的事实。

推荐：

```text
通过条件：

- 当前页面不再是登录页。
- 页面中出现“首页”、“设备”、“我的”或类似主页导航。
- 没有出现“登录失败”、“账号或密码错误”、“请输入密码”等错误提示。
- 应用没有崩溃、白屏、黑屏或长时间无响应。
```

不推荐：

```text
登录成功，页面正常。
```

### 失败条件

失败条件用于约束 agent 不要无限尝试，并帮助 agent 在异常场景下及时停止。

常见失败条件：

- 超过最大步骤仍未达到目标状态。
- 页面停留在错误页面或登录页。
- 出现明确错误提示。
- 应用崩溃、无响应、白屏、黑屏。
- 必须人工验证码、支付、人脸识别且没有可用接管流程。
- 触发了禁止行为。

### 允许行为

允许行为定义 agent 可以怎样处理非核心流程。

常见允许行为：

- 关闭广告弹窗。
- 跳过新手引导。
- 同意隐私协议。
- 允许定位、蓝牙、通知等测试所需权限。
- 等待页面加载。
- 返回上一级重新选择入口。
- 对同一操作最多重试一次。

### 禁止行为

禁止行为要写清楚，尤其是有副作用的场景。

常见禁止行为：

- 不要购买会员、下单、支付或提交真实交易。
- 不要删除、解绑、修改用户资料。
- 不要修改系统设置，除非用例明确要求。
- 不要向真实联系人发送消息。
- 不要上传隐私数据。
- 不要反复登录、反复提交表单或暴力重试。
- 不要离开被测应用执行无关任务。

## 用例质量自检清单

提交用例前检查：

- 是否写了明确测试目标。
- 是否写了包名或应用名称。
- 是否写了前置条件。
- 是否说明了已登录、未登录两种状态怎么处理。
- 是否写了测试数据来源，且没有明文敏感信息。
- 是否写了通过条件。
- 是否写了失败条件。
- 是否写了禁止行为。
- 是否设置最大步骤数和超时时间。
- 是否避免坐标、颜色、相对位置等脆弱描述。
- 是否避免让 agent 执行支付、删除、解绑、注销等高风险动作。
- 是否能单独复制给 agent 作为任务输入。
- 如果标题包含“状态、加载、展示、同步、异常、按钮状态验证”，是否没有把标题原文当成入口文案。

## 推荐的用例粒度

一个用例只验证一个主要目标。

推荐：

- 登录后进入主页。
- 进入设备页并看到添加设备入口。
- 修改昵称后页面显示新昵称。

不推荐：

- 登录、绑定设备、同步数据、修改资料、购买会员全部放在一个用例里。

如果一个流程很长，应拆成多个用例，并为每个用例明确前置条件。

## 常见反例

### 过于抽象

```text
测试首页是否正常。
```

问题：没有说明怎么进入首页，也没有说明什么叫正常。

### 过于脚本化

```text
点击坐标 500,900，输入账号，点击坐标 500,1000。
```

问题：UI 尺寸和布局变化后容易失效，也不适合 agent 泛化。

### 缺少失败边界

```text
一直尝试直到登录成功。
```

问题：可能无限重试，产生账号风控或错误状态。

### 缺少禁止行为

```text
进入设备页，完成所有能做的操作。
```

问题：可能触发解绑、删除、购买、修改资料等副作用。

### 把状态检查写成同名入口

```text
1. 打开“首页目标手表卡片与同步状态”功能。
2. 点击“我的页账号信息加载”入口。
3. 查找“首页下拉刷新与异常状态”页面。
```

问题：这些通常不是 App 中真实存在的入口。应改为进入首页或我的页后检查卡片、账号区域、刷新反馈、加载失败提示或空状态。

### 把控件交互写成同名页面

```text
1. 打开“视频点赞按钮状态验证”页面。
2. 进入“视频播放暂停与进度控制”功能。
```

问题：点赞、收藏、分享、播放暂停是视频播放页或详情页内的控件，不是独立页面。应先进入任一视频播放页，再点击目标控件并检查状态变化、面板、提示或失败原因。

### 把设备不支持当成 App 失败

```text
1. 进入手表功能页。
2. 一直查找血糖入口。
3. 找不到血糖入口则判定失败。
```

问题：血糖、血压、压力、体温、NFC、微信等能力可能依赖设备型号和固件。应写成“支持时验证；无入口、不支持、暂无数据、未连接或权限缺失时记录明确限制结果并停止”。

### 敏感流程执行过界

```text
1. 进入账号与安全。
2. 点击退出登录。
3. 点击确认退出。
```

问题：普通功能用例不应改变账号登录态。应只验证退出登录入口和二次确认弹窗，并点击取消关闭；如果误触确认，应记录为触发禁止行为。
