# AutoGLM 测试用例编写规范

## 核心原则

1. 写目标，不只写点击步骤。
2. 写状态，不依赖坐标。
3. 写允许行为，也写禁止行为。
4. 写通过条件，也写失败条件。
5. 明确遇到弹窗、登录态、加载慢、异常提示时怎么处理。
6. 敏感数据使用占位符或环境变量，不在用例中写明文密码、验证码、身份证、支付信息。
7. 用例应能独立执行，不能依赖上一条用例留下的隐含状态。
8. 测试结果以最终可观察状态为准，而不是以模型自述为准。
9. 执行步骤必须按 1、2、3、4 编号，主流程顺序必须清楚。
10. 对于设备类，要针对不同设备生成对应的测试用例，并且写清楚前置条件：不同类型设备连接下如何切换设备、未连接设备时如何连接等；设备类用例应优先从首页设备卡片或首页设备入口开始操作，首页不可达时再进入设备页。
11. 对于设备类，如何切换或连接设备应该写在前置条件或通用提示中，不应作为单独测试用例；设备连接后允许的每一个功能都应单独生成测试用例，例如微信电话、微信回复、扫码、图片回复、一键翻译、Wearfit 翻译、面对面翻译、同声传译必须分别成例。每个功能用例必须包含真实可执行的操作步骤，验证功能页面、权限、数据加载、核心交互或明确的限制提示，不能只检查入口是否存在。测试数据可使用占位符或 mock，音频、视频、图片、文件等素材类数据可以只写占位符并说明后续提供。
12. 生成的测试用例文件，应该在文件开头总结所有测试用例，给出数目和标题等。
13. 单条测试用例的标题必须使用二级标题，格式为 `## [用例ID] 用例标题`。一级标题只用于整份文件标题，不用于单条用例。
14. 用例 ID 必须以 `TC-WEARFIT` 开头，并且最后一段必须标识规则类型。规则类型只能使用不含连字符的单词：普通用例使用 `normal`，需要电脑播放音频配合手机操作的用例使用 `audio`。

## 基于代码生成用例时的要求

如果用例由人或模型阅读代码后生成，应优先从代码中提取以下信息，再写成自然语言步骤：

- 应用包名、目标 activity、页面类名或路由名。
- 页面入口，例如底部 tab、菜单项、按钮文案。
- 关键控件文案，例如“登录”、“添加设备”、“保存”。
- 关键输入框提示，例如“手机号”、“请输入密码”。
- 可用于判断结果的页面文案、resource-id、content-desc。
- 业务前置条件，例如需要登录、需要蓝牙、需要定位权限、需要真实设备。
- 高风险动作，例如支付、删除、解绑、注销、修改资料。

代码信息只能作为增强提示，不应把用例写成脆弱的坐标脚本。

推荐：

```text
1. 点击底部导航中的“设备”。
   目标状态：进入设备页，页面中出现“添加设备”或设备列表。
```

不推荐：

```text
1. 点击坐标 520,1810。
```

如果代码中能确认 activity 或 resource-id，可以写进预期结果的可观察状态中：

```text
可观察状态：

- 当前 activity 应为设备页相关 activity。
- 页面中应出现 resource-id 为 `com.example:id/add_device` 的入口，或出现文案“添加设备”。
```

## 音频相关步骤写法

涉及语音、录音、翻译、同声传译等能力时，测试步骤仍然要按用户可理解的功能流程来写，不要把规则机制写进用例。用例只描述“要完成一次语音输入/翻译/识别流程”，音频播放、长按、录音等环境配合由测试环境处理。

音频相关步骤要避免以下说法：

```text
等待自动规则执行“播放测试音频并长按识别键”。
音频素材无需提供，后续由测试环境处理。
不点击开始录音或翻译。
```

这些说法会让模型不知道当前到底要不要执行语音流程，也容易把规则当成测试终点。

推荐写法：

```text
3. 在一键翻译页面执行一次语音输入。
   目标状态：页面进入录音、识别或翻译处理中状态；测试环境会提供标准音频输入。

4. 等待翻译结果返回。
   目标状态：页面展示翻译文本、识别文本、失败提示或明确的重试入口。

5. 检查翻译结果页面。
   目标状态：页面没有崩溃、白屏或卡死，且能看到翻译结果或明确失败原因。
```

如果用例只验证入口是否能打开，不测试音频输入，则不要把用例标记为 `audio`，也不要写“音频素材”相关测试数据。步骤应明确停止在页面打开后的状态：

```text
3. 检查一键翻译页面。
   目标状态：页面出现语言选择、录音按钮、翻译说明、权限提示或连接状态。

4. 返回手表主页。
   目标状态：已回到手表功能页或设备主页。
```

如果用例需要测试真实音频输入，则必须在步骤中继续写“执行一次语音输入”和“检查结果”，不要让音频动作成为最后一步。

## 字段说明

### 用例ID

用例 ID 必须稳定，并且必须和二级标题写在同一行。固定格式：

```text
TC-WEARFIT-[模块]-[序号]-[规则类型]
```

规则类型说明：

- `normal`：普通用例，不需要特殊外部动作。
- `audio`：需要电脑播放音频，并配合手机长按、录音或语音识别的用例。

示例：

```text
TC-WEARFIT-LOGIN-001-normal
TC-WEARFIT-DEVICE-002-normal
TC-WEARFIT-VOICE-001-audio
```

单条用例标题格式必须是：

```markdown
## TC-WEARFIT-LOGIN-001-normal 手机号密码登录后进入主页
```

不要频繁修改 ID。用例管理和缺陷关联会依赖它。

### 优先级

- P0：核心链路，失败会阻断主流程，例如启动、登录、进入首页、核心设备连接。
- P1：重要功能，失败影响主要体验，但不一定阻断所有流程。
- P2：边缘场景、兼容性、文案、低频入口。

### 测试目标

测试目标回答“为什么执行这个用例”。不要只写“测试登录功能”，要写清楚验证点。

推荐：

```text
验证用户使用手机号和密码登录后，可以成功进入应用主页，并且不再停留在登录页。
```

不推荐：

```text
测试登录。
```

### 前置条件

前置条件要写 agent 无法可靠推断的信息，例如：

- 是否已经安装应用。
- 网络是否需要开启。
- 账号是否有效。
- 是否允许已登录状态直接通过。
- 是否需要蓝牙、定位、通知权限。
- 是否需要外接设备或后端环境。

对于登录态，用例必须明确：

```text
如果已经登录并进入主页，可以直接检查主页状态并判定通过，不需要退出登录后重新登录。
```

或者：

```text
本用例必须从未登录状态开始。如果当前已登录，应先停止并标记为前置条件不满足。
```

### 测试数据

敏感数据必须使用占位符：

```text
${WEARFIT_PHONE}
${WEARFIT_PASSWORD}
${TEST_VERIFY_CODE}
```

不要写：

```text
手机号：153...
密码：...
```

如果验证码、短信、人脸、支付密码等无法自动获取，应在前置条件或失败条件中说明需要人工接管。

### 执行步骤

执行步骤是直接喂给 agent 的核心内容。必须按 1、2、3、4 编号，表达清楚主流程顺序。

每个步骤建议包含四类信息：

- 动作：agent 需要做什么。
- 目标状态：做完后应该看到什么。
- 可接受变体：如果页面已经处于目标状态，应如何处理。
- 失败出口：如果无法完成，应记录什么原因。

推荐结构：

```text
1. 打开 Wearfit Pro。
   目标状态：应用启动完成，出现登录页、主页或启动过渡页。

2. 判断当前登录状态。
   如果已经在主页，直接进入第 6 步检查主页。
   如果在登录页，继续第 3 步。

3. 在登录页输入测试手机号和密码。
   如果有用户协议勾选框，先勾选用户协议。
   目标状态：账号和密码已填写，登录按钮可点击。

4. 点击“登录”。
   如果点击后没有反应，可以等待 3 秒并重试一次。
   目标状态：页面开始跳转或出现加载状态。

5. 处理登录后的过渡页面。
   如果出现权限弹窗，选择允许。
   如果出现隐私协议弹窗，选择同意。
   如果出现广告或新手引导，关闭或跳过。

6. 检查是否进入主页。
   目标状态：页面中出现“首页”、“设备”、“我的”或类似主页导航。
   停留在主页，不要继续进入无关功能。
```

不推荐：

```text
打开 Wearfit Pro。
如果当前在登录页，输入测试账号和密码，勾选用户协议后点击登录。
如果出现权限弹窗，选择允许。
如果出现隐私协议弹窗，选择同意。
如果出现广告或新手引导，关闭或跳过。
登录成功后停留在应用主页，不要继续进入无关功能。
如果出现登录失败、账号密码错误、网络错误、应用崩溃、白屏或黑屏，停止并记录失败原因。
```

原因是所有动作挤在一段里，agent 容易漏步骤，也不利于执行过程按步骤回放和排查。

每一步都要避免依赖坐标或相对位置。

不推荐：

```text
1. 点击右下角按钮。
2. 点击第二个输入框。
3. 点击蓝色按钮。
```

推荐：

```text
1. 点击文本为“手机号”或占位提示为“请输入手机号”的输入框。
2. 输入测试手机号。
3. 点击文本为“密码”或占位提示为“请输入密码”的输入框。
4. 输入测试密码。
5. 点击文本为“登录”的按钮。
```

原因是 UI 位置和样式容易变化，9B 模型也可能对“第二个”理解不稳定。

### 预期结果

预期结果描述最终应观察到的状态。尽量写成可检查的事实。

推荐：

```text
通过条件：

- 当前页面不再是登录页。
- 页面中出现“首页”、“设备”、“我的”或类似主页导航。
- 没有出现“登录失败”、“账号或密码错误”、“请输入密码”等错误提示。
- 应用没有崩溃、白屏、黑屏或长时间无响应。
```

不推荐：

```text
登录成功，页面正常。
```

### 失败条件

失败条件用于约束 agent 不要无限尝试，并帮助 agent 在异常场景下及时停止。

常见失败条件：

- 超过最大步骤仍未达到目标状态。
- 页面停留在错误页面或登录页。
- 出现明确错误提示。
- 应用崩溃、无响应、白屏、黑屏。
- 必须人工验证码、支付、人脸识别且没有可用接管流程。
- 触发了禁止行为。

### 允许行为

允许行为定义 agent 可以怎样处理非核心流程。

常见允许行为：

- 关闭广告弹窗。
- 跳过新手引导。
- 同意隐私协议。
- 允许定位、蓝牙、通知等测试所需权限。
- 等待页面加载。
- 返回上一级重新选择入口。
- 对同一操作最多重试一次。

### 禁止行为

禁止行为要写清楚，尤其是有副作用的场景。

常见禁止行为：

- 不要购买会员、下单、支付或提交真实交易。
- 不要删除、解绑、修改用户资料。
- 不要修改系统设置，除非用例明确要求。
- 不要向真实联系人发送消息。
- 不要上传隐私数据。
- 不要反复登录、反复提交表单或暴力重试。
- 不要离开被测应用执行无关任务。

## 面向 9B Agent 的写作规则

### 主流程必须编号

推荐：

```text
1. 打开应用。
2. 判断是否已登录。
3. 如果未登录，完成登录。
4. 进入设备页。
5. 检查设备页是否显示“添加设备”入口。
```

不推荐：

```text
打开应用，如果没登录就登录，然后去设备页看看有没有添加设备。
```

9B 模型对长句中的多个动作容易漏执行。编号步骤也方便定位模型在哪一步跑偏。

### 句子要短

推荐：

```text
如果看到权限弹窗，点击允许。
如果看到隐私协议，点击同意。
如果看到广告弹窗，关闭弹窗。
```

不推荐：

```text
处理可能出现的各种弹窗。
```

### 明确当前状态已经满足目标时怎么办

推荐：

```text
如果当前已经在设备页，直接检查设备页状态，不要返回首页重新进入。
```

### 明确失败时是否重试

推荐：

```text
如果点击登录后没有反应，可以等待 3 秒并重试一次。不要连续重试超过一次。
```

### 避免相对位置描述

不推荐：

```text
点击下面那个按钮。
点击第二个 tab。
点击右侧的小图标。
```

推荐：

```text
点击文本为“登录”的按钮。
点击底部导航中的“设备”。
点击页面中的“添加设备”入口。
```

### 不要让模型猜业务规则

不推荐：

```text
完成设备绑定流程。
```

推荐：

```text
进入设备页后，点击“添加设备”。如果没有可用设备或需要真实蓝牙设备，停止并记录“缺少可绑定设备”，不要尝试解绑已有设备。
```

## 用例完整示例

### 普通用例示例

```markdown
## TC-WEARFIT-LOGIN-001-normal 手机号密码登录后进入主页

### 基本信息

- 优先级：P0
- 应用名称：Wearfit Pro
- 包名：com.wakeup.howear
- 测试类型：冒烟 / 回归
- 关联模块：登录
- 最大步骤数：20
- 超时时间：120 秒

### 测试目标

验证用户可以使用手机号和密码登录 Wearfit Pro，并成功进入应用主页。

### 前置条件

- 设备已安装 Wearfit Pro。
- 网络可用。
- 测试账号有效。
- 如果当前已经登录并进入主页，可以直接检查主页状态并判定通过，不需要退出登录。

### 测试数据

- 手机号：${WEARFIT_PHONE}
- 密码：${WEARFIT_PASSWORD}

### 执行步骤

1. 打开 Wearfit Pro。
   目标状态：应用启动完成，出现登录页、主页或启动过渡页。

2. 判断当前是否已经登录。
   如果当前已经在主页，直接进入第 7 步检查主页状态，不要退出登录。
   如果当前在登录页，继续第 3 步。

3. 在登录页输入测试手机号。
   目标状态：手机号输入框中已填写 `${WEARFIT_PHONE}`。

4. 在登录页输入测试密码。
   目标状态：密码输入框中已填写 `${WEARFIT_PASSWORD}`。

5. 如果页面有用户协议勾选框，勾选用户协议。
   如果协议已经勾选，直接继续第 6 步。

6. 点击“登录”。
   如果点击后没有反应，可以等待 3 秒并重试一次。
   目标状态：页面开始跳转、出现加载状态或进入主页。

7. 处理登录后的过渡页面。
   如果出现权限弹窗，点击允许。
   如果出现隐私协议弹窗，点击同意。
   如果出现广告弹窗或新手引导，关闭或跳过。

8. 检查最终是否停留在应用主页。
   目标状态：页面中出现“首页”、“设备”、“我的”或类似主页导航。
   不要进入设备绑定、会员购买或个人资料编辑页面。

### 通用异常处理

- 如果出现登录失败、账号密码错误、网络错误、应用崩溃、白屏、黑屏或无响应，停止并记录失败原因。
- 如果需要验证码且无法自动完成，停止并记录“需要验证码接管”。
- 如果超过最大步骤仍未进入主页，停止并记录“超过最大步骤”。

### 预期结果

通过条件：

- 当前页面不再是登录页。
- 页面中出现“首页”、“设备”、“我的”或类似主页导航。
- 没有出现“登录失败”、“账号或密码错误”、“请输入密码”等错误提示。
- 应用没有崩溃、白屏、黑屏或长时间无响应。

可观察状态：

- 登录前应能识别登录页或已登录主页。
- 点击登录后应出现加载、跳转或主页。
- 最终页面应能识别主页导航或主页核心内容。

### 失败条件

- 超过最大步骤仍未进入主页。
- 点击登录后仍停留在登录页。
- 出现账号密码错误、登录失败、网络错误等提示。
- 应用崩溃、白屏、黑屏或无响应。
- 需要验证码且无法自动完成。

### 允许行为

- 可以同意隐私协议。
- 可以允许定位、蓝牙、通知等系统权限。
- 可以关闭广告弹窗或跳过新手引导。
- 可以等待页面加载。
- 登录按钮无响应时可以重试一次。

### 禁止行为

- 不要退出已登录账号。
- 不要进入会员购买、支付、设备解绑、账号注销或个人资料编辑流程。
- 不要反复点击登录超过一次重试。
- 不要向外部联系人发送消息或上传隐私数据。
```

## 用例质量自检清单

提交用例前检查：

- 是否写了明确测试目标。
- 是否写了包名或应用名称。
- 是否写了前置条件。
- 是否说明了已登录、未登录两种状态怎么处理。
- 是否写了测试数据来源，且没有明文敏感信息。
- 是否写了通过条件。
- 是否写了失败条件。
- 是否写了禁止行为。
- 是否设置最大步骤数和超时时间。
- 是否避免坐标、颜色、相对位置等脆弱描述。
- 是否避免让 agent 执行支付、删除、解绑、注销等高风险动作。
- 是否能单独复制给 agent 作为任务输入。

## 推荐的用例粒度

一个用例只验证一个主要目标。

推荐：

- 登录后进入主页。
- 进入设备页并看到添加设备入口。
- 修改昵称后页面显示新昵称。

不推荐：

- 登录、绑定设备、同步数据、修改资料、购买会员全部放在一个用例里。

如果一个流程很长，应拆成多个用例，并为每个用例明确前置条件。

## 常见反例

### 过于抽象

```text
测试首页是否正常。
```

问题：没有说明怎么进入首页，也没有说明什么叫正常。

### 过于脚本化

```text
点击坐标 500,900，输入账号，点击坐标 500,1000。
```

问题：UI 尺寸和布局变化后容易失效，也不适合 agent 泛化。

### 缺少失败边界

```text
一直尝试直到登录成功。
```

问题：可能无限重试，产生账号风控或错误状态。

### 缺少禁止行为

```text
进入设备页，完成所有能做的操作。
```

问题：可能触发解绑、删除、购买、修改资料等副作用。
